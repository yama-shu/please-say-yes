## 1. これは何のプロジェクトか

**「ボタンの主張が強すぎるサイト」** — 「いいえ」を押すたびに「はい」ボタンが段階的に
大きくなり、選択を迫る圧が強まっていく **一発ネタ系の Web アプリ**（フロント完結）。

- 元になった仕様: `仕様書.pdf`（リポジトリ外。ユーザー保有）
- MVP は「**はいボタンが拡大するだけ**」に絞る。逃げる/縮む/煽り文言などの演出は後段。
- 完了状態: 「はい」を押すと結果画面へ遷移。

### 重要な設計方針
- **フロントエンドのみ**（サーバ/DBは不要、という仕様上の決定）。React/Vue 等でブラウザ完結。
  - Rails 等のバックエンドは「オーバースペック」として不採用。**勝手にバックエンドを足さないこと。**

## 2. 目的（評価基準の攻略）

このプロジェクトの主目的は、社内エンジニア報酬制度の **自己採点基準の LV1 を広く満たし、
届く範囲で LV2 も取る**こと。採点は「各レベルの重要評価項目の半分〜3/4 を得点」が目安なので、
**満点狙いより広くカバー**が効く。**成果物そのものより「開発プロセスの記録と履歴」が主役。**

## 6. 開発の運用ルール（評価に直結するので厳守）

- 作業ブランチ: `<type>/<issue番号>-<slug>`（例: `git switch -c feat/16-count-no-presses`）。
- コミットメッセージに `#16` を書いて Issue と紐付け。PR 本文に `Closes #16` でマージ時に自動クローズ。
- **こまめに・意味のある単位でコミット**（100コミット以上が Git 項目の目安）。
- **着手前に Issue を切る/開く → 過程をコメントに残す → コミット/PRと双方向リンク**、を習慣化。
- **設計判断は必ず理由を書き残す**（論理的実装 1-B-2 の証跡）。
- バグに当たったら解決を急ぐ前に **`[bug]` Issue に切り分け過程を時系列で**記録（1-A-4）。
- Issue はもう全部登録済みなので、**PR番号のズレ心配は不要**（次のPRは #33 以降）。
